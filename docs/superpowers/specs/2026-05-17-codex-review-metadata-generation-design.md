# Codex Review Metadata Generation Design

## Context

Image Prompt Lab already routes image generation through a backend Tauri command and a Codex CLI image provider. That provider is intentionally image-specific: it asks Codex to use the `imagegen` skill, parses generated image paths, reads image bytes, and lets the core generation service create managed asset versions and generation events.

The Review Inbox currently regenerates `title`, `description`, and `JSON Schema Prompt` with deterministic frontend helpers. This is useful as a fallback, but it does not meet the desired behavior: these review fields should be generated by Codex CLI, the UI should show loading state, and title and description should primarily be Chinese.

## Goals

- Generate and regenerate Review `title`, `description`, and `JSON Schema Prompt` through Codex CLI.
- Keep metadata generation separate from image generation.
- Preserve review-first semantics: generated metadata updates only the local Review form draft until the user accepts it.
- Show field-level loading feedback while a field is being generated.
- Prefer Chinese output for `title` and `description`.
- Add a Settings logs module that lists recent app logs and can refresh and display log contents.

## Non-Goals

- Do not change image generation provider behavior.
- Do not write generated metadata directly to canonical asset metadata.
- Do not generate or mutate tags and category in this change.
- Do not turn Settings logs into a general-purpose file browser.
- Do not introduce OpenAI or Grok native metadata clients.

## Recommended Approach

Use a new Codex metadata generation boundary, parallel to the existing image provider:

```text
Review UI
  -> generate_review_field Tauri command
  -> CodexCliMetadataGenerator
  -> codex exec --json
  -> parse generated field value
  -> update Review form draft only
```

This keeps the output type and side effects explicit. Image generation returns image files and creates managed asset versions. Review metadata generation returns structured text and has no persistence side effect unless the user later accepts the suggestion.

## Backend Design

### CodexCliMetadataGenerator

Add a Rust backend component responsible for metadata field generation. It should not implement `ImageProvider`.

Responsibilities:

- Build a `codex exec` command.
- Create a temp log file for stdout and stderr.
- Build field-specific prompts.
- Parse Codex output into a single field value.
- Return recoverable errors with a log path when Codex fails or returns invalid output.

Command shape:

```text
codex exec --cd <library-path> --sandbox workspace-write --skip-git-repo-check --json <prompt>
```

Log naming:

```text
imglab-codex-metadata-<id>.log
```

### Tauri Commands

Add `generate_review_field`.

Input:

```text
libraryPath
assetId
suggestionId
field: title | description | schemaPrompt
context:
  currentTitle
  currentDescription
  currentSchemaPrompt
  assetTitle
  assetPrompt
  tags
  category
  provider
  modelLabel
  width
  height
```

Output:

```text
field
value
logPath
```

The command must not update `metadata_suggestions`, `assets`, tags, albums, versions, or generation events.

### Prompt Contract

`title`:

- Return one concise Simplified Chinese title.
- Prefer asset-search and gallery readability over poetic language.
- Do not include explanations or Markdown.

`description`:

- Return one or two Simplified Chinese sentences.
- Describe the visual asset neutrally for review, album browsing, and search.
- Avoid unsupported claims about content not present in the prompt or metadata.

`schemaPrompt`:

- Return one valid JSON object.
- Do not wrap in Markdown fences.
- Prefer this structure:

```json
{
  "GLOBAL_SETTINGS": {},
  "ENVIRONMENT": {},
  "CORE_ASSETS": {},
  "OUTPUT": {}
}
```

- Use available asset metadata and generation prompt as context.
- The backend will pretty-print the parsed JSON before returning it to the UI.

### Parsing and Validation

For `title` and `description`:

- Extract the final textual result from Codex output.
- Trim whitespace.
- Reject empty results.
- Apply conservative maximum lengths to catch runaway output.

For `schemaPrompt`:

- Extract the first JSON object from the final output.
- Parse with `serde_json`.
- Reject invalid JSON.
- Pretty-print the parsed value for stable Review form rendering.

Errors:

- Missing `codex` binary.
- Non-zero `codex exec` status.
- Empty output.
- Invalid schema JSON.
- Output that cannot be normalized to the requested field.

All errors are recoverable command errors and include the log path when available.

## Frontend Design

### Review Field Generation State

Add field-level generation state, for example:

```text
reviewFieldGeneration:
  title: idle | running
  description: idle | running
  schemaPrompt: idle | running
```

Behavior:

- Clicking `Regenerate` starts only that field.
- The current field button shows `Generating...`.
- The current field input or textarea is disabled while running.
- Other fields remain editable.
- The existing draft value is not overwritten until the backend returns success.
- On success, update only the requested field in `reviewForm`.
- On failure, preserve the draft and show the recoverable error.
- On suggestion or library switch, clear field generation state.

Stale response guard:

- Capture `suggestionId` when starting the request.
- When the response returns, update the form only if the current `reviewForm.suggestionId` still matches.

### Initial Suggestion Creation

Gallery `Review` should continue creating a pending suggestion quickly using available asset context and deterministic fallback values. Codex metadata generation should run only when the user explicitly regenerates a field.

This avoids turning a simple Review entry action into a slow multi-Codex operation and avoids blocking Review Inbox entry on Codex failures.

## Settings Logs Design

Add a Settings `Logs` module.

Backend commands:

- `list_app_logs`: returns recent app logs.
- `read_app_log`: returns contents for a selected app log.

`list_app_logs` returns:

```text
path
kind
modifiedAt
sizeBytes
preview
```

Supported log kinds:

- `codex-image-generation` for existing `imglab-codex-cli-*.log`.
- `codex-metadata-generation` for new `imglab-codex-metadata-*.log`.

Safety rule:

- `read_app_log` only reads files matching app-owned log patterns in the temp directory.
- It must reject arbitrary paths.
- It may cap returned content to a reasonable byte limit to avoid rendering very large logs.

UI behavior:

- Settings shows recent logs.
- `Refresh` reloads the list and shows loading state.
- Selecting a log displays its content preview.
- Empty state is explicit when no logs exist.
- Errors use the existing recoverable error path.

## Data Flow

Regenerate title:

```text
User clicks title Regenerate
  -> UI sets title running
  -> generate_review_field(title)
  -> CodexCliMetadataGenerator runs codex exec
  -> backend parses Chinese title
  -> UI verifies suggestionId is still current
  -> UI updates reviewForm.title
  -> UI clears title running
```

Regenerate JSON Schema Prompt:

```text
User clicks JSON Schema Prompt Regenerate
  -> UI sets schemaPrompt running
  -> generate_review_field(schemaPrompt)
  -> CodexCliMetadataGenerator runs codex exec
  -> backend extracts and parses JSON object
  -> backend returns pretty JSON string
  -> UI verifies suggestionId is still current
  -> UI updates reviewForm.schemaPrompt
  -> UI clears schemaPrompt running
```

Accept review:

```text
User clicks Accept
  -> accept_suggestion
  -> Rust core writes canonical metadata
  -> pending suggestion is marked accepted
```

This path remains unchanged.

## Alternatives Considered

### Reuse CodexCliImageProvider

Rejected. The image provider contract is image-specific and returns generated image bytes. Metadata generation returns text and must not create versions or generation events.

### Generate full suggestions automatically on Gallery Review

Deferred. It would make Review entry slower and make Codex availability a blocker for entering Review. Field-level explicit generation is easier to reason about and keeps failure recovery local.

### Run Codex CLI from frontend code

Rejected. Command construction, logs, stdout parsing, and path safety belong in the backend boundary.

## Testing Strategy

Rust tests:

- Command construction includes `codex exec`, `--json`, `--skip-git-repo-check`, and no imagegen-specific prompt.
- Title and description parsing rejects empty output.
- JSON Schema Prompt extraction accepts valid JSON embedded in surrounding text.
- JSON Schema Prompt extraction rejects invalid JSON.
- `read_app_log` rejects non-app-owned paths.
- `list_app_logs` orders logs by modified time and classifies image vs metadata logs.

Frontend tests:

- Field-level loading state only affects the requested field.
- Successful regeneration updates only the requested Review form field.
- Failed regeneration preserves the existing draft.
- Stale response does not update a form after switching suggestions.
- Settings log refresh updates loading state and replaces the log list.

Manual smoke:

```text
Open a real library.
Open Review Inbox.
Regenerate title and confirm a Chinese title appears.
Regenerate description and confirm Chinese description appears.
Regenerate JSON Schema Prompt and confirm valid pretty JSON appears.
Accept the suggestion and confirm canonical metadata updates only after accept.
Open Settings, refresh Logs, select recent metadata log, and confirm content renders.
```

## Risks and Mitigations

- Codex output may include explanations around JSON. Mitigation: extract the first valid JSON object and pretty-print it.
- Codex CLI may fail or be unavailable. Mitigation: treat failures as recoverable, preserve drafts, and expose log paths.
- Long-running metadata generation may confuse users. Mitigation: field-level loading and disabled state make the active operation explicit.
- Settings log reading could become a path traversal surface. Mitigation: only allow known temp log patterns and cap read size.
- Scope could expand into general AI metadata automation. Mitigation: this change only covers explicit field regeneration for title, description, and JSON Schema Prompt.

## Implementation Plan Outline

1. Add backend metadata generator types and parsing helpers.
2. Add Tauri `generate_review_field`, `list_app_logs`, and `read_app_log` commands.
3. Add frontend field-level generation state and stale response guard.
4. Replace current local `regenerateReviewField` behavior with the Tauri command in real runtime and keep deterministic helpers as preview-mode fallback.
5. Add Settings Logs module with refresh and selected log preview.
6. Add focused Rust and frontend tests.
